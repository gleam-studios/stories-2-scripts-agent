# 11 大纲规范

## 一、适用文件

1. `01_<剧名>_英文大纲.docx`
2. `05_<剧名>_中文大纲.docx`

## 二、Quick Reference 必须包含

1. 项目名
2. Logline
3. 目标市场
4. 类型
5. 集数
6. 单集时长
7. 核心情绪/视觉标签
8. 主角速览

## 三、主体内容结构（章节顺序）

Word / Markdown 正文必须按以下顺序输出，缺一不可：

| 章节 | 内容 |
|------|------|
| **Adaptation Positioning** | 改编定位：核心钩子、受众定位、差异化方向 |
| **Season Outline** | 全季前提、主矛盾、4 阶段推进、大反转节点、终局兑现 |
| **Episode Outline** | 每集独立条目（格式见第四节），必须覆盖全部集数 |
| **Season Hook Map** | 全季钩子推进总览（4 段弧的钩子方向，对齐 `adaptation_plan.season_hook_map`） |
| **Episode Hook Map** | 每集集末钩子表（对象表，非散文列表） |
| **Paywall Map** | 卡点规划表（对象表，与 `adaptation_plan.paywall_map` 对齐） |

## 四、每集条目固定字段（Episode Outline 格式）

每集必须包含以下全部字段，均为一句话标准，不扩写：

| 字段 | 说明 |
|------|------|
| `episode_id` | 如 `EP01`，与 `adaptation_plan` 一致 |
| `title` | 本集标题，简短有力 |
| `arc_phase` | 阶段枚举，**固定四选一**：`launch` / `rise` / `storm` / `finale` |
| `logline` | 本集一句话剧情定位 |
| `core_event` | 本集最重要的叙事事件 |
| `turn` | 本集的核心转折（信息、关系或权力变化） |
| `emotional_goal` | 本集观众情绪任务（一句话：观众到片尾应感受到什么） |
| `relationship_shift` | 主线关系在本集的变化（一句话：谁和谁之间发生了什么位移） |
| `antagonist_pressure` | 反派/机构在本集的具体一步（一句话：威胁或压制动作） |
| `satisfaction_beat` | 本集的爽点兑现（一句话，对应 `rules/06_爽点矩阵` 5 类中的某一类） |
| `end_hook` | 集末钩子（一句话，对应 `hook_type`） |

### `arc_phase` 枚举定义

| 枚举值 | 对应阶段 | 比例参考 |
|--------|---------|---------|
| `launch` | 起势段 | 前 0%-20% |
| `rise` | 攀升段 | 20%-50% |
| `storm` | 风暴段 | 50%-80% |
| `finale` | 决战段 | 最后 20% |

与 `adaptation_plan.rhythm_curve` 和 `rules/05_节奏曲线` 的阶段词一致。

### `satisfaction_beat` 标签参考

来自 `rules/06_爽点矩阵`，从以下 5 类中选一个标签标注，再写一句具体内容：

- `status_reversal`（身份碾压）
- `face_slap`（打脸复仇）
- `underdog_rise`（逆袭翻盘）
- `emotional_eruption`（情感爆发）
- `truth_drop`（悬念揭秘）

## 五、Hook Map 与 Paywall Map 格式

### Episode Hook Map（每集一条，对象格式）

```
| episode_id | hook_type | hook_out |
```

- `hook_type`：使用以下词汇之一：`identity_tease` / `evidence_drop` / `alliance_shift` / `betrayal_reveal` / `romance_peak` / `cliffhanger` / `reversal` / `setup`
- `hook_out`：一句话说明集末悬念内容

必须与 `adaptation_plan.episode_hook_map` 对齐。

### Paywall Map（卡点，对象格式）

```
| episode_id | position_ratio | phase | type | beat | viewer_task |
```

- `position_ratio`：在全季中的位置比例（如 `0.20`）
- `phase`：`first_lock` / `mid_lock` / `storm_lock` / `finale_lock`
- `type`：5 类卡点类型之一（参见 `rules/04_商业化卡点策略`）
- `beat`：一句话说明卡点内容
- `viewer_task`：卡点处观众悬而未决的问题（一句话，驱动续播）

必须与 `adaptation_plan.paywall_map` 字段对齐。

## 六、与 `adaptation_plan` 的对齐要求

大纲必须可追溯到改编规划，具体要求：

1. 每集的 `arc_phase` 必须与 `adaptation_plan.season_arc` 的阶段区间对应。
2. `Episode Hook Map` 的 `episode_id` 与 `hook_type` 必须与 `adaptation_plan.episode_hook_map` 一致。
3. `Paywall Map` 的所有卡点集号与 `position_ratio` 必须与 `adaptation_plan.paywall_map` 一致。
4. `satisfaction_beat` 的标签类型必须在 `adaptation_plan.satisfaction_mix` 中有对应的设计方向。

## 七、强制要求

1. 大纲只做结构，不写逐场正文。
2. 每集必须有 `EPxx` 标识，`episode_outlines` 数量必须等于 `quick_reference.episode_count`。
3. 卡点数量与 `rules/04_商业化卡点策略` 的密度建议对齐。
4. `arc_phase` 必须使用枚举值，禁止自由书写。

## 八、禁止项

1. 禁止把大纲写成完整剧本。
2. 禁止写镜头、景别、机位或制作备注。
3. 禁止把设定集内容整段搬进大纲。
4. 禁止用"以此类推"或"格式同上"替代任何一集的条目。
5. 禁止 Hook Map 或 Paywall Map 只写散文描述，必须为可对表的结构化格式。

## 九、输出前自审

```
□ episode_outlines 条目数 = quick_reference.episode_count（无遗漏）
□ 每集条目包含全部 11 个字段（无缺失）
□ 所有 arc_phase 使用枚举值：launch / rise / storm / finale
□ 每集 satisfaction_beat 标签来自 5 类枚举
□ Episode Hook Map 与 adaptation_plan.episode_hook_map 完全对应
□ Paywall Map 与 adaptation_plan.paywall_map 完全对应
□ Word 正文包含全部 6 个章节（含 Season Hook Map、Episode Hook Map、Paywall Map）
```
