# 11 大纲规范

## 一、定位

大纲是整个项目的**主控文档**。

剧本、分镜表、设定集都以大纲为起点展开。大纲必须把以下所有内容总结到位，让下游三个文件各自深化，而不是自己重新发明。

## 二、适用文件

1. `01_<剧名>_英文大纲.docx`
2. `05_<剧名>_中文大纲.docx`

## 三、Word / Markdown 章节顺序（固定，缺一不可）

| 序号 | 章节名 | 作用 |
|------|--------|------|
| 1 | Quick Reference | 项目元信息 |
| 2 | Core Selling Points | 多维卖点（对接制片/投流） |
| 3 | Adaptation Positioning | 改编定位与差异化方向 |
| 4 | World Setting Summary | 世界观摘要（设定集起点） |
| 5 | Character Cards | 主要人物轻量卡（剧本/设定集起点） |
| 6 | Main Plot | 主线结构 |
| 7 | Subplot Map | 副线设计 |
| 8 | Overall Story Structure | 叙事骨架（开端→发展→中段反转→高潮→结局） |
| 9 | Episode Outline | 分集梗概（全集，每集 11 字段） |
| 10 | Season Hook Map | 全季钩子推进总览 |
| 11 | Episode Hook Map | 每集集末钩子表 |
| 12 | Paywall Map | 卡点规划表 |
| 13 | Signature Scenes | 高能名场面清单 |
| 14 | Viral Clip Candidates | 切片传播桥段清单 |

---

## 四、Quick Reference

必须包含：

1. 项目名
2. Logline
3. 目标市场
4. 类型
5. 集数
6. 单集时长
7. 核心情绪/视觉标签
8. 主角速览

---

## 五、Core Selling Points（核心卖点）

必须从以下 4 个维度各写 1-2 句：

| 维度 | 说明 |
|------|------|
| `genre_hook` | 类型卖点：这部剧属于什么类型、核心吸引力是什么 |
| `emotional_hook` | 情绪卖点：观众追下去是为了爽/甜/虐/上瘾中的哪种情绪 |
| `character_hook` | 人物卖点：主角或关键人物有什么让人记得住的特质 |
| `differentiation` | 差异化卖点：区别于同类项目的独特之处是什么 |

---

## 六、Adaptation Positioning（改编定位）

必须包含：

- `core_hook`：一句话说明改编后的核心钩子
- `audience_positioning`：受众定位
- `similarity_avoidance_direction`：如何与原著/同类作品区别开

---

## 七、World Setting Summary（世界观设定摘要）

**现实题材也必须写**（至少写行业背景与社会规则）；有架空/系统/奇幻/特殊规则的题材必须详写。

必须包含：

| 字段 | 说明 |
|------|------|
| `era_and_environment` | 时代与环境（现代都市/古代王朝/近未来/架空世界观等） |
| `social_rules` | 该世界的核心社会规则或权力结构（谁有权、谁无权、靠什么争） |
| `industry_or_special_setting` | 行业背景或特殊设定（如：直播行业、冰球名校、宫廷礼制、系统任务规则） |
| `key_constraints` | 对叙事有影响的硬约束（如：穿越后身份不能暴露；系统任务失败有惩罚；证据必须在公开场合才有效） |

**注意**：摘要即可，完整版在 `setting_bible` 里展开。

---

## 八、Character Cards（主要人物设定卡）

每个主要角色一张卡，必须包含以下字段（一句话标准，不写人物小传）：

| 字段 | 说明 |
|------|------|
| `character_id` | 与 setting_bible 和 adaptation_plan 对齐，如 CHAR-01 |
| `name` | 姓名（含别名） |
| `role_type` | 枚举：`protagonist` / `antagonist` / `ally` / `romance` / `catalyst` / `supporting` |
| `identity` | 在故事世界里的身份（职业/社会地位） |
| `personality_keywords` | 3-5 个性格关键词 |
| `desire_goal` | 他/她最想得到什么（一句话） |
| `relation_to_protagonist` | 与主角的关系（一句话） |
| `contrast_point` | 人物反差点——让他/她立体起来的矛盾性（一句话） |
| `arc_note` | 全季成长或反转轨迹（一句话） |

**至少必须有**：主角、主要对手、情感线角色、推动剧情的关键配角。

---

## 九、Main Plot（主线结构）

必须清楚回答以下 4 个问题（每个 1-2 句）：

| 字段 | 问题 |
|------|------|
| `protagonist_want` | 主角想得到什么？ |
| `antagonist_block` | 谁或什么在阻碍他/她？用什么方式？ |
| `escalation_path` | 主线冲突如何逐步升级？（3-4 个关键节点） |
| `resolution_direction` | 主线最终如何解决？ |

---

## 十、Subplot Map（副线设计）

每条副线一个条目，必须包含：

| 字段 | 说明 |
|------|------|
| `subplot_id` | 如 SUB-01 |
| `subplot_type` | 枚举：`romance` / `revenge` / `identity_secret` / `family` / `power_struggle` / `investigation` / `competition` / `friendship` |
| `characters_involved` | 涉及的角色 ID 列表 |
| `arc_summary` | 这条副线从开始到结束发生了什么（2-3 句） |
| `function_in_story` | 它为主线做了什么（是撑情绪/撑节奏/撑人物/提供信息）（一句话） |

**原则**：副线不能抢主线；副线必须和主线有至少一次交织点。

---

## 十一、Overall Story Structure（整体叙事骨架）

按 5 段叙事阶段写，每段必须包含：

### 11.1 Opening（开端）
- 主角出场方式与初始处境
- 引爆事件（什么事情打破了平衡）
- 第一个钩子（观众被什么留下来）

### 11.2 Development（发展段）
- 主角开始如何行动
- 矛盾如何逐步升级
- 关系如何在这一阶段变化
- 爽点/误会/危机的连续出现节奏

### 11.3 Midpoint Reversal（中段反转）
- 哪个真相在这里露出一部分
- 主角遭遇的重大打击或身份/处境变化
- 对手在这一阶段如何阶段性占优

### 11.4 Climax（后段高潮）
- 终极冲突如何爆发
- 最大秘密在哪里揭晓
- 人物关系如何决裂或重组

### 11.5 Ending（结局）
- 主线如何解决
- 主要人物命运落地
- 情绪如何收尾
- 是否留第二季钩子（选填）

---

## 十二、Episode Outline（分集梗概）

每集一个条目，必须包含以下 11 个字段（一句话标准）：

| 字段 | 说明 |
|------|------|
| `episode_id` | 如 EP01 |
| `title` | 本集标题 |
| `arc_phase` | 枚举：`launch` / `rise` / `storm` / `finale` |
| `logline` | 本集一句话定位 |
| `core_event` | 本集最重要的叙事事件 |
| `turn` | 本集核心转折（信息/关系/权力变化） |
| `emotional_goal` | 本集观众情绪任务 |
| `relationship_shift` | 主线关系在本集的变化 |
| `antagonist_pressure` | 反派/机构在本集的具体一步 |
| `satisfaction_beat` | 爽点标签（5 类枚举）+ 一句具体内容 |
| `end_hook` | 集末钩子（hook_type 枚举 + 一句内容） |

`episode_outlines` 数量必须等于 `quick_reference.episode_count`，不得遗漏任何一集。

---

## 十三、Season Hook Map

全季 4 阶段的钩子方向总览，每段一条：

```
| phase | ratio_range | hook_direction |
```

---

## 十四、Episode Hook Map

每集一行，结构化表格：

```
| episode_id | hook_type | hook_out |
```

`hook_type` 枚举：`identity_tease` / `evidence_drop` / `alliance_shift` / `betrayal_reveal` / `romance_peak` / `cliffhanger` / `reversal` / `setup`

与 `adaptation_plan.episode_hook_map` 对齐。

---

## 十五、Paywall Map

卡点表格：

```
| episode_id | position_ratio | phase | type | beat | viewer_task |
```

`phase` 枚举：`first_lock` / `mid_lock` / `storm_lock` / `finale_lock`

与 `adaptation_plan.paywall_map` 对齐。

---

## 十六、Signature Scenes（高能名场面）

全季中必须拍好的 **3-8 个** 标志性场面，每条一个条目：

| 字段 | 说明 |
|------|------|
| `scene_id` | 如 SIG-01 |
| `episode_id` | 属于哪一集 |
| `scene_type` | 枚举：`public_reversal` / `identity_reveal` / `romance_peak` / `confrontation` / `betrayal` / `sacrifice` / `truth_drop` |
| `description` | 这场戏发生了什么（2-3 句，写得让人想看） |
| `why_unmissable` | 为什么这场戏是全剧关键（对情节/人物/观众情绪的作用，一句话） |
| `production_weight` | 枚举：`highest` / `high` / `medium`（给分镜和 AI 生产定优先级） |

---

## 十七、Viral Clip Candidates（切片传播桥段）

全季中天然适合做切片或短视频传播的 **3-6 个** 桥段，每条一个条目：

| 字段 | 说明 |
|------|------|
| `clip_id` | 如 CLIP-01 |
| `episode_id` | 属于哪一集 |
| `clip_type` | 枚举：`status_reversal` / `face_slap` / `romance_moment` / `plot_twist` / `emotional_eruption` / `confrontation` |
| `description` | 这个桥段的内容（1-2 句，写什么让人想转发） |
| `hook_in_3s` | 前 3 秒能看到什么让人停下来（一句话） |

---

## 十八、强制要求

1. 大纲只做结构，不写逐场正文。
2. 每集必须有 `EPxx` 标识，数量等于 `episode_count`。
3. `arc_phase` 必须使用枚举值，禁止自由书写。
4. `satisfaction_beat` 标签必须来自 5 类爽点枚举。
5. 卡点数量与 `rules/04_商业化卡点策略` 的密度建议对齐。
6. `Signature Scenes` 和 `Viral Clip Candidates` 必须存在，最少各 3 条。

## 十九、禁止项

1. 禁止把大纲写成完整剧本。
2. 禁止写镜头、景别、机位或制作备注。
3. 禁止把设定集内容整段搬进大纲（摘要即可，完整版在设定集）。
4. 禁止用"以此类推"替代任何一集的条目。
5. 禁止 Hook Map / Paywall Map / Signature Scenes / Viral Clips 只写散文，必须为结构化格式。

## 二十、输出前自审

```
□ 14 个章节全部存在，顺序正确
□ episode_outlines 条目数 = quick_reference.episode_count
□ 每集 11 个字段均已填写
□ 所有 arc_phase 使用枚举值
□ Character Cards 覆盖主角 / 对手 / 情感线 / 关键配角
□ Subplot Map 至少 2 条副线
□ Overall Story Structure 5 段均已写完
□ Signature Scenes 至少 3 条
□ Viral Clip Candidates 至少 3 条
□ Episode Hook Map 与 adaptation_plan.episode_hook_map 对应
□ Paywall Map 与 adaptation_plan.paywall_map 对应
```
